# Plan: kenspc Plugin v3.0.2 — Over-Constraint Cleanup

## Objective

把 v3.0.0 引入但实际上违反 v3 自身 bitter-lesson 哲学的两条「正向约束」撤掉：

1. 强制英文输出（"summaries / progress messages in English only"）— 4 个
   agent .md（plan-/guide-/task-document-reviewer + task-implementer）+
   项目 `CLAUDE.md` + plugin `README.md` + plugin `.claude-plugin/plugin.json`
   description 字段中的 "English-only output" 子句 + v3 plan AC6 释义。
   `code-fixer.md` 的 code-artifacts 英文约束保留不动。
2. Planned Dispatch 表的 `Status` 列（永远停在 `pending` 的静态制品）—
   5 个 dispatching SKILL.md + v3 plan AC8 反向化。

scope 限于 plugin v3.0.2 patch；不动 dispatch 后的 result table、不动
generate-brief、不动 `shared/discovery-framework.md` 中文教学例句、不动
全局 user-private CLAUDE.md。

`plugin.json` 升 `3.0.1 → 3.0.2`。

## Background

**v3.0.0 自身的 Non-Goals 第 7 条已记录这次清理的根因**：
> "Live updating" dispatch tables (Claude Code's TUI already handles this)

也就是说 v3.0.0 plan 当时就承认 TUI 才是真正的 live state，但仍然在每个
dispatching skill 加了带 `Status` 列的静态表。表的语义（`pending` →
`done`）和 markdown 的字面 / 不可回写本质冲突 — Claude 的 tool-use channel
和文本输出 channel 是分离的，dispatch 一旦发出，主上下文无法回头编辑先前
渲染的表格。结果就是：**用户看到的表永远停在 `pending`**。

CLI 现在底栏有 live agents-list，让这张静态表既冗余又误导；VS Code Claude
Code 暂时没有等同 UI，所以那张表是用户的唯一线索 — 它在那种环境下撒谎，
比没有更糟。

强制英文输出违反同一哲学的另一面 — 用 SKILL/agent 文本去约束 Claude 的
*运行时输出语言决策*，与 v3.0.0 删 `ULTRATHINK`、把推理深度交给 `effort:`
frontmatter 是同类问题：把语言决策交回给上下文判断。

v3.0.1 的 CHANGELOG（L46–54）刚刚**加固**了 AC6 / AC8 这两条 guard。本次
是反向调整，但属于"修正 v3 自身的执行偏差"，不是反复改主意 — bitter-lesson
不是"少加 guard"，而是"guard 应 enforce 真正有效的事"。

## Design Decisions

### D1 — AC6 删除 vs 反向化

删除。AC6 enforce 的是"输出语言"，而正确做法是不约束输出语言（让上下文
判断），既不正向也不反向 — 不存在"必须双语"这种合理反向语义。

### D2 — AC8 反向化（不删除）

反向化。Planned Dispatch 表本身是合法的 — 它向用户传达 *intent*（"接下来
会派发哪几个 agent"），尤其在 VS Code 没有底栏的环境下保留可见性。错的是
`Status` 列让它假装是 *status* 表。

新 AC8 文本：

```bash
# Every dispatching skill has a Planned Dispatch list/table that names all
# agents it will dispatch, but does NOT have a Status column or "pending"
# marker masquerading as live state.
for f in \
  plugins/kenspc/skills/generate-plan/SKILL.md \
  plugins/kenspc/skills/generate-guide/SKILL.md \
  plugins/kenspc/skills/generate-task/SKILL.md \
  plugins/kenspc/skills/task-review/SKILL.md \
  plugins/kenspc/skills/task-implement/SKILL.md ; do
  grep -qiE 'Planned Dispatch|即将派发' "$f" || { echo "MISSING list: $f"; continue; }
  # No Status column header in the Planned Dispatch table window.
  # (Result tables in Schema A/E still have Status columns; those live in
  #  separate sections — bounded by the awk window cutoff.)
  awk '/Planned Dispatch|即将派发/,/^##|^---$/' "$f" \
    | grep -qE '\| *Status *\|' \
    && echo "STATUS COLUMN STILL PRESENT in Planned Dispatch: $f"
done | tee /tmp/ac8.log
test ! -s /tmp/ac8.log
```

`awk` window 从 "Planned Dispatch" / "即将派发" 标题取到下一个 `##` 或
`---`，确保只检查 Planned Dispatch 那张表，不波及 result table。

### D3 — Planned Dispatch 新表头

`# / Agent / Role`（也接受 `# / Agent / Purpose`）。引导句：
"Planned Dispatch — 即将派发以下 agents:" 或纯英文 "Planned Dispatch:"
（沿用各 SKILL 既有语言基调）。

每行的 `Role` 文本就是该 agent description 字段第一句话的精炼，例如：

- `requirements-reviewer` → "Reviews completeness against requirements"
- `task-implementer` → "Implements all incomplete tasks in batch"

### D4 — 版本号 v3.0.2 vs v3.1.0

选 **v3.0.2 patch**。

理由：

- 不引入新 SKILL 接口、不变更 agent 名称、不改变 dispatch 数量、不改
  CONTEXT block schema。
- 被移除的两项（强制英文输出 / 状态列）是 v3.0.0 的*执行偏差*，不是
  *功能特性*。修正错误属于 patch 语义。
- README 移除一条 "feature" 卖点 — 可见性影响低（这条卖点本身就是误述）。

### D5 — 不动 dispatch 后的 result table

result table（task-review 的 Schema A 表、generate-plan 的 Schema E 表
等）保留所有 `Status` 列。它们是 dispatch 完成*之后*才渲染的，能正确填
真实 status，不存在撒谎问题。
