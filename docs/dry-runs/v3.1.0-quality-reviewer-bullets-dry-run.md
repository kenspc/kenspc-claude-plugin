## Notes on gate behavior observed

- Condition 2 of the drive-by bullet (mechanically forced) carried the
  weight in two of three "adjacent-code modification" hunks. Without
  Condition 2, both would have been false-positive flagged.
- Condition 3 of the over-engineering bullet (boundary validation) was
  not strictly load-bearing in this dry-run because Condition 1 already
  excluded the boundary-validating script — but it provides a second
  line of defense if a future change is genuinely scope-extending while
  also doing boundary validation.
- Condition 1 of both bullets (not in task requirements) is the
  cheapest exclusion when the reviewer has the task document — which is
  the standard `task-review` mode. Bare-`changes` mode would lose
  Condition 1 and rely more on Conditions 2 and 3, so future tightening
  candidates would focus there if false-positives surface in bare-mode
  usage.
