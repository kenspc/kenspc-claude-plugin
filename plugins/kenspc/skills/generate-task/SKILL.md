# Generate Task Document

Decompose a plan document into fine-grained, directly executable tasks by
reading the actual codebase. Three phases: Analyze, Confirm, Verify.

## Trigger Phrases

Use this skill when the user explicitly asks to **create a task document
from a plan**, using phrases like: "generate tasks", "break down tasks",
"create task document", "decompose the plan", "拆任务", "生成任务",
"任务分解", "帮我拆任务", or invokes `/kenspc-task` directly.

Avoid triggering this skill when the user:
- Asks to create a plan (use generate-plan instead).
- Asks to implement tasks (use task-implement instead).
- Wants to discuss task priorities without generating a document.
- Asks a question about an existing task document.

## Quality bar

A useful task document covers every plan Implementation Step, gives each
task a single clear objective and concrete acceptance criteria (no vague
language like "as appropriate", "if needed", "properly"), and orders tasks
so that no task starts before its dependencies are DONE.

## Prerequisites

- A plan document generated by generate-plan (or following the same structure).
- A project with actual code to analyze.

## Arguments

$ARGUMENTS format: PLAN_PATH [PHASE] [CUSTOM_INSTRUCTIONS]

- PLAN_PATH: path to the plan document (e.g., docs/plans/notification-plan.md).
- PHASE: optional, specifies which phase(s) to decompose (e.g., "Phase 1",
  "Phase 1 and 2"). If omitted, decompose the entire plan.
- CUSTOM_INSTRUCTIONS: optional free-text that narrows scope or adds constraints.

If no arguments are provided, ask the user for the plan document path.

## Phase 1: Analyze

**Goal**: produce a candidate task list grounded in the actual code, with
explicit dependencies and concrete acceptance criteria for each task.

**Inputs**: PLAN_PATH (and optional PHASE / CUSTOM_INSTRUCTIONS); the
project's CLAUDE.md, README, and config files; the project's source tree.

**DONE when**:
- Every plan Implementation Step in scope is covered by at least one task.
- Each task has a single clear objective and concrete, verifiable acceptance
  criteria.
- Task dependencies form a valid DAG (no circular dependencies).
- Files each task creates or modifies are identified up front.
- Task sizing is S or M; L tasks have been broken further; XL is rejected.
- Each task touches a manageable scope of files (around 8 or fewer is the
  default; more requires explicit justification).

**Constraints**:
- **Why deep code reading matters**: decomposition without reading code
  misses hidden dependencies and existing implementations. Read the public
  interfaces of relevant modules — function signatures, exports, API routes,
  data models, type definitions — not every line of implementation.
- **Cross-phase dependency rule**: for tasks in the earliest in-scope phase,
  base analysis on existing code. For tasks in later phases that depend on
  earlier phases' output, base analysis on the plan's description of what
  those earlier phases will produce, and annotate each such task with
  `Depends on: Task N`.
- If the plan document does not exist, cannot be parsed, or has no
  Implementation Steps section, ask the user for the correct path.
- If a plan step's technical approach contradicts the existing codebase
  after reading the code, this is a plan-level issue. Stop and inform the
  user — it is not solvable at the task level.

### Task Sizing Guidelines

| Size | Files | Scope | Example |
|------|-------|-------|---------|
| **XS** | 1 | Single function or config change | Add a validation rule |
| **S** | 1-2 | One component or endpoint | Add a new API endpoint |
| **M** | 3-5 | One feature slice | User registration flow |
| **L** | 5-8 | Multi-component feature | Search with filtering and pagination |
| **XL** | 8+ | **Too large — break it down further** | — |

Target S and M tasks. If a task is L or larger, break it into smaller tasks.

## Phase 2: Confirm

**Goal**: get the user's confirmation on the task list, then write the
task document at the right path.

**Inputs**: candidate task list from Phase 1; project's documentation
conventions (CLAUDE.md or `docs/tasks/` default); reference format at
`${CLAUDE_PLUGIN_ROOT}/references/task-document-example.md`.

**DONE when**:
- The user has confirmed the task list (or requested adjustments have been
  applied and re-presented).
- The task document is written at the resolved path with each task
  including: number + title, description with files to create/modify,
  concrete acceptance criteria, `Status: TODO`, and `Depends on: Task N`
  where applicable.
- Cross-phase dependency note is included at the top of the document if
  any task has a `Depends on` annotation.

**Presentation format** (Phase 2 Step 1):

```
Found N tasks from [plan name] [phase scope]:

1. Task 1: [title] — [N] files
   Criteria: [one-sentence acceptance criteria summary]
2. Task 2: [title] — [N] files, depends on Task 1
   Criteria: [one-sentence acceptance criteria summary]
...

Confirm, or adjust tasks before writing?
```

**Output path resolution** (priority order):
1. CLAUDE.md-specified task document location.
2. `docs/tasks/` (create if missing).

**Filename**: derive the base name from the plan filename by removing the
`.md` extension and any trailing `-plan` suffix. Examples:
`notification-plan.md` → `notification`; `auth-strategy.md` → `auth-strategy`.
- Entire plan: `{base-name}-tasks.md` (e.g., `notification-tasks.md`).
- Specific phase: `{base-name}-phase{N}-tasks.md` (e.g.,
  `notification-phase1-tasks.md`).
- Multiple phases: `{base-name}-phase{N}-{M}-tasks.md`.

**Conflict check**: if writing a phase-specific file when a full-plan task
file already exists for the same plan (or vice versa), ask the user:
overwrite, create alongside, or cancel.

## Phase 3: Verify via review agent

**Goal**: run the task document through `task-document-reviewer` and present
the consolidated review summary.

**Inputs**: path of the task document just written; PLAN_PATH; project root
path (or "N/A" if not in a project).

### Step 1: Render Planned Dispatch table

Before invoking the review agent, render this table so the user sees the
planned dispatch:

| # | Agent | Role |
|---|-------|------|
| 1 | task-document-reviewer | Reviews task document and plan alignment |

### Step 2: Construct CONTEXT block and dispatch

Build the structured CONTEXT block:

```
CONTEXT
- TASK_DOC_PATH: <path of the task document just written>
- SOURCE_PATH: <plan document path from $ARGUMENTS>
- PROJECT_PATH: <project root path, or "N/A" if not in a project>
```

Tell the user: "Task document written to [path]. Dispatching review agent
now."

Then dispatch a subagent using the Agent tool:
- Agent name: `task-document-reviewer`
- description: "Review task document"
- prompt: the CONTEXT block above

### Step 3: Render the result table (Schema E) and present the summary

When the subagent returns, render the result table verbatim from the
agent's output — Schema E:

| Angle | Status     | Changes       | Commit  |
|-------|------------|---------------|---------|
| 1     | PASSED     | —             | —       |
| 2     | FIXED (n)  | section X, Y  | def5678 |
| 3     | NOTED      | open question | ghi9012 |
| 4     | PASSED     | —             | —       |

Below the table, present:
- The Changes prose (per FIXED / NOTED row: what changed, why, commit hash).
- Any `## Plan-Level Concerns` section the agent produced (these surface
  upstream-plan issues the user should resolve at the plan level, not the
  task level).
- Any unresolved issues.
