# Generate Task Document

Decompose a plan document into fine-grained, directly executable tasks by reading
the actual codebase. Three phases: Analyze, Confirm, Verify.

## Trigger Phrases

Use this skill when the user explicitly asks to **create a task document from a plan**,
using phrases like: "generate tasks", "break down tasks", "create task document",
"decompose the plan", "拆任务", "生成任务", "任务分解", "帮我拆任务",
or invokes `/kenspc-task` directly.

**Do NOT trigger this skill** when the user:
- Asks to create a plan (use generate-plan instead)
- Asks to implement tasks (use task-implement instead)
- Wants to discuss task priorities without generating a document
- Asks a question about an existing task document

## Common Rationalizations

| Agent says | Why it's wrong |
|---|---|
| "Plan 已经够细了，直接执行" | If the user invoked generate-task, the granularity is not sufficient. Read the code and decompose properly. Do not skip. |
| "代码太多读不完，按 plan 描述拆就行" | Decomposition without reading code will miss hidden dependencies and existing implementations. Phase 1 Step 2 deep code reading is mandatory. |
| "只有 2 个任务，不需要让用户确认" | The number of tasks does not determine whether confirmation is needed. The user needs to verify granularity and ordering are reasonable. Phase 2 cannot be skipped. |
| "实现的时候再决定顺序" | Implementation order matters. Wrong order means rework. Task dependencies must be explicit before execution begins. |
| "任务很明显，不用写出来" | Unwritten tasks hide dependencies and edge cases. Explicit tasks survive context window compaction. |

## Red Flags

Stop and inform the user if any of these occur (thresholds are starting values — adjust based on project experience):

- A single plan phase decomposes into more than ~15 tasks → Granularity may be too fine, or the phase itself should be split. Inform the user.
- After reading the code, a plan step's technical approach contradicts the existing codebase → This is a plan-level issue, not solvable at task level. Stop and inform the user.
- More than ~30% of tasks cannot have concrete acceptance criteria → The plan is not specific enough. Suggest the user revisit the plan.

## Prerequisites

- A plan document generated by generate-plan (or following the same structure)
- A project with actual code to analyze

## Arguments

$ARGUMENTS format: PLAN_PATH [PHASE] [CUSTOM_INSTRUCTIONS]

- PLAN_PATH: path to the plan document (e.g., docs/plans/notification-plan.md)
- PHASE: optional, specifies which phase(s) to decompose (e.g., "Phase 1",
  "Phase 1 and 2"). If omitted, decompose the entire plan.
- CUSTOM_INSTRUCTIONS: optional free-text that narrows scope or adds constraints

If no arguments are provided, ask the user for the plan document path.

## Phase 1: Analyze

### Step 1: Read the plan

Read the plan document at PLAN_PATH. If a specific PHASE was requested, identify
the relevant Implementation Steps for that phase only. If no phase was specified,
read all Implementation Steps.

If the plan document does not exist, cannot be parsed, or has no Implementation Steps
section, ask the user for the correct path.

### Step 2: Deep code reading

Two layers of analysis:

**Project level:**
- Read CLAUDE.md (project and root level) for conventions, tech stack, constraints
- Scan the project structure (directory listing)
- Read key config files (package.json, *.csproj, docker-compose.yml, .env.example, etc.)

**Task level:**
- For each module or area mentioned in the plan's Implementation Steps, locate the
  actual source files in the project
- Read the public interface of each relevant module: function signatures, exports,
  API routes, data models, type definitions
- Do NOT read every line of implementation — focus on interfaces and contracts
- Identify which files need to be created vs modified

**Cross-phase dependency rule:**
- For tasks in Phase 1 (or the earliest phase): base analysis entirely on existing code
- For tasks in later phases that depend on earlier phases' output: base the analysis
  on the plan's description of what those earlier phases will produce, not on current
  code (which doesn't exist yet). Annotate these tasks with their dependencies.

### Step 3: ULTRATHINK decomposition

Synthesize the plan's Implementation Steps with the code analysis. For each step:
1. Break it into tasks where each task has a single clear objective and is independently
   verifiable
2. Determine execution order based on dependencies
3. Write acceptance criteria for each task — concrete and verifiable, no vague language
   ("as appropriate", "if needed", "properly", "sufficient" are not acceptable)
4. Identify which files each task will create or modify
5. For tasks depending on earlier phases' output, add a `Depends on: Task N` annotation

### Task Sizing Guidelines

| Size | Files | Scope | Example |
|------|-------|-------|---------|
| **XS** | 1 | Single function or config change | Add a validation rule |
| **S** | 1-2 | One component or endpoint | Add a new API endpoint |
| **M** | 3-5 | One feature slice | User registration flow |
| **L** | 5-8 | Multi-component feature | Search with filtering and pagination |
| **XL** | 8+ | **Too large — break it down further** | — |

Target S and M tasks. If a task is L or larger, break it into smaller tasks.

### Pre-presentation checklist

Before presenting to user, verify:
- [ ] Every plan Implementation Step is covered by at least one task
- [ ] Every task has concrete acceptance criteria (no vague language)
- [ ] Task dependencies form a valid DAG (no circular dependencies)
- [ ] No task touches more than ~8 files

## Phase 2: Confirm

### Step 1: Present task list

Present the decomposed tasks to the user for confirmation. For each task, show:
- Task number and title
- Acceptance criteria summary (one sentence)
- Files involved (create/modify)
- Dependencies (if any)

Format:
```
Found N tasks from [plan name] [phase scope]: / 从 [plan name] [phase scope] 分解出 N 个任务：

1. Task 1: [title] — [N] files
   Criteria: [one-sentence acceptance criteria summary]
2. Task 2: [title] — [N] files, depends on Task 1
   Criteria: [one-sentence acceptance criteria summary]
...

Confirm, or adjust tasks before writing? / 确认，或在写入前调整任务？
```

Wait for user confirmation. If the user requests changes (reorder, merge, split,
remove tasks), apply them and re-present.

### Step 2: Determine output path and check conflicts

**Output directory priority:**
1. If CLAUDE.md specifies a task document location, use it
2. Otherwise, use `docs/tasks/`
3. Create the directory if it does not exist

**Filename:**
Derive the base name from the plan filename by removing the file extension and
any trailing `-plan` suffix (e.g., `notification-plan.md` → `notification`,
`auth-strategy.md` → `auth-strategy`).
- Entire plan: `{base-name}-tasks.md` (e.g., `notification-tasks.md`)
- Specific phase: `{base-name}-phase{N}-tasks.md` (e.g., `notification-phase1-tasks.md`)
- If the phase covers multiple phases: `{base-name}-phase{N}-{M}-tasks.md`

**Conflict check before writing:**
- If writing a phase-specific file, check if a full-plan task file already exists for
  the same plan (e.g., writing `notification-phase2-tasks.md` when `notification-tasks.md`
  exists)
- If writing a full-plan file, check if phase-specific task files already exist
- If conflict found, ask the user: overwrite, create alongside, or cancel

### Step 3: Write the task document

Read the reference format at `${CLAUDE_PLUGIN_ROOT}/references/task-document-example.md`.
Write the task document following that format.

Each task must include:
- Task number and title
- Description (what to do, which files to create/modify)
- Acceptance criteria (specific, verifiable)
- Status: TODO
- Dependencies (if any): `Depends on: Task N`

If cross-phase dependencies exist, add a note at the top of the document:
```
> Tasks marked with "Depends on" assume prior tasks have been completed.
> Execute tasks in order.
```

## Phase 3: Verify via review agent

After the task document is written, automatically launch a review cycle.
Do not wait for user instruction.

### Step 1: Read and render the review prompt

Read the file `prompts/review.md` from this skill's directory.

Replace all placeholders:
- `{{TASK_DOC_PATH}}` — path of the task document just written
- `{{SOURCE_PATH}}` — the plan document path from $ARGUMENTS
- `{{PROJECT_PATH}}` — project root path, or "N/A" if not in a project

### Prompt variables

| Variable | Source | Values |
|----------|--------|---------|
| {{TASK_DOC_PATH}} | Phase 2 Step 3 | Path of the written task document |
| {{SOURCE_PATH}} | $ARGUMENTS PLAN_PATH | Plan document path |
| {{PROJECT_PATH}} | Project root | Path or "N/A" |

### Step 2: Dispatch the review agent

Tell the user:
"Task document written to [path]. Dispatching review agent now. /
任务文档已写入 [path]。正在启动审查代理。"

Dispatch a subagent using the Agent tool:
- prompt: the rendered review prompt
- description: "Review task document"

### Step 3: Present results

When the subagent returns, present its summary. The summary includes:
- Which review angles passed
- Every change made with the reason (and git commits)
- Any plan-level concerns noted (these require user attention at the plan level)
- Any unresolved issues
